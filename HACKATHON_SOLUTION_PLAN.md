## 2. Какие типы данных здесь есть и зачем они нам

### 2.1 Самые ценные источники для MVP

1. `share/Выгрузки/дочерние предприятия/Employes/*.txt`  
   Это лучший стартовый набор для детекторов. Внутри уже есть:
   - ФИО
   - дата рождения
   - паспортные данные
   - адрес
   - телефон
   - email
   - СНИЛС
   - табельные номера

   Примеры файлов:
   - `anketa_dms.txt`
   - `soglasie_na_obrabotku_pd.txt`
   - `raspiska_poluchenie_noutbuka.txt`
   - `doverennost_na_poluchenie_posylki.txt`

   Эти файлы надо использовать как первый "золотой" набор для отладки правил и отчетов.

2. `share/Выгрузки/Логистика/*.csv`  
   Здесь много строк и понятная схема. Особенно важны:
   - `customers.csv`: есть ФИО клиентов;
   - `logistics.csv`: есть адреса доставки;
   - `sales.csv`, `stores.csv`, `products.csv`: mostly опорный бизнес-контекст, полезен для join-логики и отрицательных примеров.

3. `share/Выгрузки/дочерние предприятия/Billing/*`  
   Здесь лежат наиболее "корпоративные" structured-данные:
   - `subscribers.csv`: много идентификаторов и статусов;
   - `plan.json`: тарифные планы, сами по себе не ПДн, но полезны как отрицательный пример;
   - `client.parquet`, `company.parquet`, `physical.parquet`: по именам это, вероятно, основные таблицы с клиентами, юрлицами и физлицами.

   Без поддержки `parquet` решение не будет полноценно закрывать ТЗ.

### 2.2 Шумные, но важные источники

1. `share/Прочее`  
   Это не просто "архив PDF". Там есть как минимум три разных класса файлов:
   - публичные документы, где ПДн почти нет или они вторичны;
   - шаблоны документов с полями под ПДн;
   - реально рискованные документы (`xls` с доходами/декларациями, отдельные формы, сканы).

   Это критично: детектор должен различать
   - заполненный документ с ПДн
   - шаблон с пустыми полями `ФИО ____`, `адрес ____`

   Иначе будет очень много ложных срабатываний.

2. `share/Выгрузки/Сайты` и `share/Документы партнеров`  
   Это отличный стресс-тест на false positive:
   - privacy policy
   - rules
   - mediakit
   - terms of service
   - страницы LiveJournal
   - служебные HTML вроде `away` и captcha page

   Здесь много слов "email", "phone", "privacy", "personal data", но это не значит, что файл надо поднимать в `УЗ-1/УЗ-2`.

3. `share/Архив сканы` и `share/Прочее/*.tif`  
   Это OCR-контур. Его не нужно включать в самый первый baseline на все подряд, но его обязательно нужно закладывать во второй этап.

### 2.3 Отдельный риск-блок

`share/Прочее/*.xls` с названиями вроде:

- `Сведения о доходах ...`
- `Декларация ...`
- `Главный бухгалтер сведения ...`

Это потенциально высокорисковые файлы с финансовыми и личными данными. Их надо выделить в отдельный приоритетный срез после `txt/csv/json/parquet`.

## 3. Какие "программы" и технические артефакты уже есть

## 4. Основные выводы для стратегии решения

### 4.1 Что нужно делать обязательно

- строить pipeline от формата файла, а не один общий regex-скан по байтам
- отдельно обрабатывать structured и unstructured данные
- валидировать числовые идентификаторы
- учитывать контекст вокруг совпадений
- считать не только количество матчей, но и тип источника
- уметь различать:
  - реальные ПДн
  - шаблон полей
  - публичный текст о политике обработки ПДн

### 4.2 Что не стоит делать в первой итерации

- не надо начинать с OCR на весь корпус
- не надо начинать с `mp4`
- не надо пытаться решать задачу через LLM-first
- не надо считать любое слово "здоровье", "биометрия", "политика" автоматическим `УЗ-1`

## 5. Рекомендуемое поэтапное решение

## Этап 0. Зафиксировать целевую архитектуру и метрики

Цель:
- определить формат итогового CLI-решения
- зафиксировать уровни УЗ
- определить контрольный набор файлов для ручной проверки

Результат:
- `README` с архитектурой
- конфиг категорий ПДн и порогов
- список контрольных файлов из:
  - `Employes/*.txt`
  - `Логистика/*.csv`
  - `Billing/*`
  - 15-20 шумных PDF/HTML из `Прочее`, `Сайты`, `Документы партнеров`

## Этап 1. Собрать extraction layer для "быстрых побед"

Приоритет форматов:
1. `txt`
2. `csv`
3. `json`
4. `html`
5. `docx`
6. `pdf`
7. `xls`
8. `parquet`
9. `tif/png/jpg/gif`
10. `doc`
11. `mp4`

Почему так:
- `txt/csv/json/html` дадут быстрый рабочий baseline;
- `parquet` обязателен по ТЗ, но требует зависимостей;
- `ocr/video` дорогие по времени и должны идти после нормального текстового контура.

Минимальные зависимости для production-версии:
- `pypdf` или `pdfminer.six`
- `python-docx`
- `beautifulsoup4` + `lxml`
- `openpyxl` и/или `xlrd`
- `pyarrow` или `duckdb` для `parquet`
- `pandas` только если действительно нужен, не как обязательный core
- `Pillow` + `pytesseract` для OCR
- `python-magic` или хотя бы fallback на `mimetypes/file`

Важно: в текущей среде этих библиотек нет, поэтому их установку надо включить в первый рабочий спринт.

## Этап 2. Построить schema-aware детекторы для structured data

Это самый выгодный этап по качеству.

Нужно сделать:
- анализ по названиям колонок
- анализ по содержимому ячеек
- поддержку чанков для больших CSV
- поддержку `parquet` без полной загрузки в память

Что искать в первую очередь:
- `name`, `fio`, `customer_name`, `person`, `employee`
- `address`, `destination_address`
- `email`, `phone`
- `passport`, `snils`, `inn`
- `birth`, `dob`

Отдельно:
- `plan.json`, `products.json`, `incidents.json` должны стабильно классифицироваться как "нет ПДн" или very low risk

Ожидаемый эффект:
- сильный результат на `dataset5/dataset6`
- хорошая explainability
- мало ложных срабатываний

## Этап 3. Построить rule-based детекторы для текста

Базовые категории:

- обычные ПДн:
  - ФИО
  - телефон
  - email
  - адрес
  - дата рождения

- государственные идентификаторы:
  - паспорт РФ
  - СНИЛС
  - ИНН
  - водительское удостоверение
  - MRZ

- платежные:
  - карта + Luhn
  - БИК
  - расчетный счет
  - CVV только при очень сильном контексте

- биометрические:
  - только по узкому списку ключей и контексту

- специальные категории:
  - только при персональном контексте, а не по одиночным словам

Ключевая идея:
- regex -> normalization -> validator -> context scoring -> category

Без этого на публичных PDF снова будут ложные `УЗ-1`.

## Этап 4. Ввести анти-false-positive слой

Это критично именно для этого корпуса.

Нужны отдельные правила для:

1. Шаблонов документов  
   Признаки:
   - длинные подчеркивания
   - `ФИО ______`
   - `адрес ______`
   - `подпись ______`

   Такие документы нельзя автоматически поднимать как полноценные ПДн-файлы.

2. Публичных policy/rules/terms документов  
   Признаки:
   - много слов про `privacy`, `персональные данные`, `политика`, `правила`
   - мало валидных сущностей
   - нет устойчивых комбинаций "субъект + идентификатор + адрес"

3. Бизнес-справочников и каталогов  
   `plan.json`, `products.json`, часть `sample.md` и similar.

4. Шумных HTML  
   нужно вычищать:
   - JS/CSS
   - служебные токены
   - JSON blobs в HTML
   - рекламный/метаданный мусор

## Этап 5. OCR и image triage

Не запускать OCR на весь корпус сразу.

Правильный порядок:

1. Сначала определить, какие изображения документоподобны:
   - `tif`
   - крупные `jpg/png`
   - изображения с текстовой плотностью

2. OCR в первую очередь на:
   - `share/Архив сканы`
   - `share/Прочее/*.tif`

3. OCR на изображения из `Сайты` только выборочно  
   там много декоративных картинок и веб-ассетов.

4. Для PDF:
   - если текст извлекается нативно, OCR не нужен
   - если текст пустой или очень короткий, включать OCR fallback

## Этап 6. УЗ engine с прозрачными правилами

Предлагаю не делать магию, а зафиксировать таблицу правил.

Базовая логика:

- `УЗ-1`
  - есть специальные категории
  - есть биометрия
  - или сильная комбинация спецкатегории + идентификаторы

- `УЗ-2`
  - есть платежные данные
  - или много госидентификаторов

- `УЗ-3`
  - немного госидентификаторов
  - или большой объем обычных ПДн

- `УЗ-4`
  - только обычные ПДн в малом объеме

Но считать надо не просто "сколько regex-совпадений", а хотя бы:
- `validated_hits`
- `rows_affected`
- `distinct_entities_hash`
- `document_is_template`

## Этап 7. Отчетность

Нужны три выхода:

1. `CSV`
2. `JSON`
3. `Markdown summary`

Рекомендованный набор полей:

- путь
- формат файла
- источник/директория
- категории ПДн
- число подтвержденных находок
- число подозрительных находок
- УЗ
- confidence
- причина классификации
- признак "шаблон"
- признак "OCR used"
- ошибки чтения, если были

Отдельно в Markdown:
- сводка по директориям
- топ рискованных файлов
- статистика по категориям
- список ошибок/неподдержанных файлов

## Этап 8. Проверка качества

Для этого датасета нужна не абстрактная accuracy, а набор ручных тестовых срезов.

Надо собрать mini-goldset из:

- 10 файлов `Employes`
- 5 CSV из логистики
- 3-5 structured-файлов billing
- 10 публичных PDF/HTML, где ожидается low/no PII
- 5 шаблонов документов из `Прочее`
- 10 рискованных `xls/pdf/docx` из `Прочее`
- 10 сканов `tif`

Проверять отдельно:
- recall на явных ПДн
- false positive на public docs
- false positive на шаблонах
- корректность УЗ

## Этап 9. Финальная демонстрация

На защите лучше показывать не "мы умеем regex", а такой story:

1. mixed storage с 3000+ файлами
2. scanner сам понимает формат
3. structured data разбирается по схеме
4. public docs не взлетают в `УЗ-1`
5. кадровые и биллинговые документы ловятся корректно
6. OCR подключается только где нужен
7. итоговый отчет privacy-safe

## 8. Короткий итог

Лучший шанс на сильное решение здесь дает не "один умный детектор", а конвейер из четырех слоев:

1. правильное извлечение текста/структуры
2. валидируемые детекторы
3. анти-false-positive правила
4. прозрачный rules engine для УЗ

Реальный сильный MVP для этого набора файлов должен:

- уверенно работать на `txt/csv/json/html`
- обязательно поддерживать `parquet`
- хорошо фильтровать public docs и шаблоны
- подключать OCR выборочно
- не пытаться делать `mp4` глубоко в первой версии

Если развивать уже имеющийся baseline из ноутбука, его надо не "допилить немного", а фактически перевести из демонстрационного regex-скрипта в нормальный format-aware pipeline.
